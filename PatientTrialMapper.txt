================================================================================
                            PATIENTTRIALMAPPER 
================================================================================

WHAT IS THIS PROJECT?
----------------------
PatientTrialMapper is an AI system that automatically screens patients for
clinical trial eligibility. Given a patient's medical profile and a trial's
eligibility criteria, the model predicts whether the patient is:
  - Eligible
  - Not Eligible
  - Unknown (ambiguous — sent to human review)

The model also gives a REASON for its decision in plain English.

WHY DOES THIS MATTER?
---------------------
Clinical trial recruitment is broken. Manually reviewing patient records
against trial criteria takes doctors hours per patient. Trials fail because
they can't recruit fast enough. This project automates that bottleneck.

WHAT MAKES IT SPECIAL (FOR RESUME/INTERVIEWS)?
-----------------------------------------------
1. Fine-tuned a 7-billion parameter LLM (Mistral-7B) on medical data
2. Used QLoRA — an advanced technique to train huge models with limited GPU
3. Built a complete 3-page web app (Streamlit) around the model
4. Ran training on Kaggle's free GPU (Tesla T4 x2)
5. The model gives REASONING, not just a yes/no label

================================================================================
TECH STACK
================================================================================

Base Model     : Mistral-7B-Instruct-v0.3 (7 billion parameters)
Fine-tuning    : QLoRA (Quantized Low-Rank Adaptation)
Quantization   : 4-bit NF4 (reduces 14GB model to ~5GB GPU usage)
Training Lib   : TRL (Transformer Reinforcement Learning library)
PEFT Lib       : Parameter-Efficient Fine-Tuning (LoRA adapter management)
UI Framework   : Streamlit (Python web app)
Language       : Python 3.12
Platform       : Kaggle (free GPU for training), Windows (local UI)

BASE_MODEL = "mistralai/Mistral-7B-Instruct-v0.3"
  - The starting point. A 7B parameter instruction-following model.
  - Downloads ~14 GB from Hugging Face on first run.
  - Alternative: "microsoft/Phi-3-mini-4k-instruct" (smaller, no token needed)

HF_TOKEN = ""
  - Hugging Face API token. Needed for gated models like Mistral.
  - Get it from: huggingface.co/settings/tokens

LORA_R = 16
  - LoRA rank. Controls how many parameters are trained.
  - Higher = more capacity but more memory. 16 is standard.

LORA_ALPHA = 32
  - LoRA scaling factor. Usually 2x the rank.
  - Controls how much the adapter influences the base model.

LORA_DROPOUT = 0.05
  - Prevents overfitting during training.

LORA_MODULES = ['q_proj', 'k_proj', 'v_proj', 'o_proj', 'gate_proj', 'up_proj', 'down_proj']
  - Which layers inside the transformer to fine-tune.
  - These are the attention and feed-forward layers.

MAX_SEQ_LEN = 512
  - Maximum token length for input + output combined.
  - Patient narrative + criteria + answer must fit in 512 tokens.

MAX_STEPS = 500
  - Number of training iterations.
  - With 16 examples, this means ~31 passes through the data (epochs).

BATCH_SIZE = 2
  - Examples processed at once per GPU.

GRAD_ACC = 4
  - Gradient accumulation. Effective batch = 2 x 4 = 8.
  - Simulates larger batch without needing more GPU memory.

LEARNING_RATE = 2e-4  (0.0002)
  - How fast the model updates its weights.
  - Standard value for LoRA fine-tuning.

================================================================================
HOW THE MODEL WORKS — STEP BY STEP
================================================================================

INPUT (what goes into the model):
  Patient narrative:
    "65-year-old male with type 2 diabetes. HbA1c 7.2%, on metformin."

  Trial criteria:
    "- Adults 18-75 with type 2 diabetes
     - HbA1c between 7.0% and 10.0%
     - Not on insulin therapy"

PROMPT FORMAT (how data_prep.py formats it):
  ### Patient Profile
  65-year-old male with type 2 diabetes. HbA1c 7.2%, on metformin.

  ### Trial Eligibility Criteria
  - Adults 18-75 with type 2 diabetes
  - HbA1c between 7.0% and 10.0%
  - Not on insulin therapy

  ### Is the patient eligible?

MODEL OUTPUT (what Mistral generates):
  "Eligible. Patient meets all criteria: age 65 is within 18-75,
   HbA1c 7.2% falls within range, and metformin is not insulin."

PARSING (how predict.py extracts the answer):
  - Checks if output contains "not eligible" → label = "Not Eligible"
  - Checks if output contains "eligible" → label = "Eligible"
  - Otherwise → label = "Unknown"
  - Everything after the verdict = reasoning

================================================================================
WHAT IS QLORA? (EXPLAIN IN INTERVIEWS)
================================================================================

Normal fine-tuning: Update all 7 billion parameters → needs 4x14GB = 56GB GPU
QLoRA approach:
  Step 1 — Quantize: Compress model weights from 16-bit to 4-bit → ~5GB GPU
  Step 2 — LoRA: Don't touch the frozen 4-bit weights.
            Instead, add small trainable "adapter" matrices alongside them.
            Only 42 million parameters are trained (0.57% of 7B total).
  Step 3 — During inference: Combine frozen base + trained adapters.

Result: Train a 7B model on a single 16GB GPU in ~90 minutes.
Adapter file size: ~100MB (vs 14GB for full model)

================================================================================
ERRORS FIXED DURING DEVELOPMENT (GOOD FOR INTERVIEWS)
================================================================================

1. bitsandbytes not working on Windows
   → Train on Kaggle/Linux. Local app uses demo mode.

2. trl 1.5 API breaking changes
   → dataset_text_field and max_seq_length moved to SFTConfig
   → Pre-tokenize dataset manually to bypass SFTTrainer internals

3. BFloat16/Float16 gradient scaler conflict
   → Tesla T4 doesn't support BFloat16 natively
   → Cast all bf16 params to fp16 after LoRA application
   → Disabled fp16 training flag (4-bit quant handles memory)

4. Streamlit key/value conflict (version 1.37+)
   → Cannot use both `key=` and `value=` on same widget
   → Changed key to match session state variable name directly

5. git clone failing on Kaggle (internet off)
   → Enable internet in Kaggle Settings before running

6. os.chdir() not affecting shell commands in Jupyter
   → Use %cd magic command instead

================================================================================
INTERVIEW TALKING POINTS
================================================================================

"What did you build?"
  An end-to-end clinical NLP system. Fine-tuned Mistral-7B using QLoRA on
  patient-trial eligibility data. Built a complete Streamlit UI with single
  screening, batch processing, and model analytics dashboard.

"Why QLoRA instead of full fine-tuning?"
  Full fine-tuning a 7B model needs ~56GB GPU (4 A100s, ~$5000/hour).
  QLoRA lets us train on a single T4 GPU (free on Kaggle) in 90 minutes.
  We only train 0.57% of parameters (42M vs 7B) with no significant quality loss.

"What was the hardest part?"
  Debugging API breaking changes across trl versions (0.8 vs 1.5).
  The training loop worked but internal tokenization logic changed completely.
  Had to pre-tokenize the dataset manually to bypass SFTTrainer's new internals.

"How would you improve it?"
  1. More training data — TREC 2022 has 2000 labeled pairs
  2. Better evaluation — clinical F1 should be weighted toward recall
     (missing an eligible patient is worse than a false positive)
  3. Structured output — force model to output JSON for reliable parsing
  4. Human-in-the-loop for Unknown predictions — route to doctor review

"What are the model's limitations?"
  1. Small training set (20 examples) — metrics are not production-ready
  2. Cannot handle complex multi-condition criteria well
  3. Hallucination risk — model may fabricate plausible-sounding reasoning
  4. No uncertainty quantification — confidence is rule-based, not probabilistic
